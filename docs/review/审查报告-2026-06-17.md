# LAK-Agent 设计审查报告

- **审查对象**：政法智能知识Agent平台.md（总纲规格文档）
- **审查日期**：2026-06-17
- **审查类型**：全面审查（架构 / 安全合规 / 可靠性 / 可运维性 / 落地可行性）

---

## 一、总体评价

总纲文档定位清晰、技术栈务实、核心安全基线到位，作为项目总纲规格是合格的。以下按七个维度展开缺失项分析与改进建议。

---

## 二、架构层面

### 2.1 缺少系统分层架构图

**现状**：文档描述了 Agent 路由逻辑，但无分层架构，组件间交互关系不直观。

**建议分层**：

| 层 | 职责 | 组件 |
|----|------|------|
| 接入层 | 认证鉴权、限流、日志拦截 | API Gateway / Spring Interceptor |
| 编排层 | 意图识别、路由分发、结果合规校验 | 主Agent + 子Agent调度器 |
| 能力层 | RAG检索、对话管理、工单接口调用 | RAG引擎、会话管理器、工单适配器 |
| 数据层 | 向量存储、业务数据、会话缓存、文件存储 | Qdrant / MySQL / Redis / MinIO |

### 2.2 知识库全生命周期管理 — 完全缺失

**现状**：文档只覆盖 RAG 检索，知识从哪来、怎么入库、如何更新均未涉及。

**必须补全的内容**：
- **文档预处理流水线**：多格式解析（PDF/Word/HTML）→ 分段策略（固定长度 vs 语义分段）→ Embedding 生成 → Qdrant 写入
- **知识更新机制**：增量同步 vs 定时全量；政策文件的生效/废止标记；过期知识的下线策略
- **元数据管理**：每条知识片段的元数据 Schema（来源文件编号、生效时间、发布机构、适用范围）
- **知识质量评估**：检索命中率监控、低质量知识标记、知识库健康度报告

### 2.3 多轮对话管理 — 仅一句话

**现状**：只提到"会话记忆持久化至 Redis"，缺少完整的多轮对话设计。

**必须补全的内容**：
- **对话状态机**：`新建 → 意图识别 → 信息采集（Slot-Filling）→ 答复 → 关闭`
- **上下文窗口管理**：消息历史如何裁剪？（滑动窗口 / 摘要压缩 / 混合）
- **多轮信息补齐流程**：投诉子Agent 的"对话补齐工单必填项"需要明确定义 Slot 列表、追问策略、最大追问轮数

### 2.4 子Agent协作模式未定义

**现状**：每个子Agent独立描述，未覆盖跨Agent场景。

**必须明确的决策**：
- 是否可能出现一个请求需要跨子Agent协作？（如"某政策下如何办理某事项"）
- 若涉及协作，主Agent 串行调度还是并行调度？
- 子Agent 是否有能力调用其他子Agent？

## 三、安全合规

### 3.1 等保等级需明确目标

**现状**："满足政务等保要求"过于模糊。

**影响**：等保二级和三级在技术实现上的差距显著（见下表），必须在详细设计阶段锁定目标。

| 要求项 | 等保二级 | 等保三级 |
|--------|---------|---------|
| 身份鉴别 | 基础用户名/密码 | 双因素认证 + 口令复杂度策略 |
| 访问控制 | 粗粒度角色 | 细粒度 RBAC + 强制访问控制 |
| 安全审计 | 基础日志 | 审计日志防篡改 + 审计记录备份 |
| 通信加密 | 建议 | 强制 TLS 1.2+ |
| 数据加密 | 建议 | 敏感数据必须加密存储 |
| 漏洞扫描 | 定期 | 实时 + 定期扫描 |

### 3.2 缺少认证授权设计

**必须明确的内容**：
- 用户角色定义（政法干警 / 外包人员 / 管理员 / 审计员）
- 认证方式（统一身份认证平台对接？本地账号？SSO？）
- 权限粒度（角色级 RBAC 还是知识库级别细粒度控制？）
- API 鉴权方案（JWT / OAuth2 / Session Token？）

### 3.3 密钥与配置安全管理

- 百炼平台 API Key 等敏感凭证如何管理？（环境变量 / Vault / KMS？）
- 数据库密码、Redis 密码如何加密存储？
- 敏感配置是否支持运行时动态刷新（不重启）？

### 3.4 审计日志防篡改

- 日志不可删除/不可修改的技术保障
- 审计日志的存储与备份策略
- 审计日志查询权限控制

## 四、可靠性

### 4.1 单点故障风险分析

| 组件 | 故障影响 | 建议 |
|------|---------|------|
| Qdrant | 全部 RAG 不可用 | 数据备份 + 快速恢复方案；评估是否可降级为关键词检索 |
| Redis 7 | 全部进行中会话丢失 | Redis AOF 持久化 + Sentinel；评估会话重建的可行性 |
| MySQL 8.0 | 审计日志/工单写入失败 | 日志写入失败的降级策略（本地文件缓冲 → 恢复后补写） |
| 百炼 API | 大模型服务不可达 | **核心风险**：需定义降级方案 |

### 4.2 大模型服务降级链

```
百炼API调用超时（如>30s）
  → 重试1次
    → 仍失败：
      政策/办事类 → "系统繁忙，请稍后重试" + 落审计日志
      投诉类     → 降级为纯表单收集，标记"待人工补发"
```

**建议**：评估是否需要部署本地备用小模型（如 Qwen 开源版）作为极端情况下的兜底。

### 4.3 缺少限流与防刷设计

文档提到 Redis 限流，但未展开：
- 限流维度（IP? 用户? 接口?）
- 限流算法（令牌桶 / 滑动窗口？）
- 触发限流后的响应策略

## 五、Agent 与提示词设计

### 5.1 Prompt 管理策略

**现状**：未提及 Prompt 如何存储和变更管理。

**建议**：政法场景合规要求高，推荐：
- Prompt 存储在配置中心（Nacos / Apollo / 本地 YAML 文件），不与业务代码耦合
- Prompt 变更走 Git 版本控制 + 审批流程
- 支持模板变量（动态注入用户身份、时间、机构信息）

### 5.2 缺少 RAG 评估体系

Agent 质量需要量化，建议定义：

| 指标 | 说明 | 目标值（参考） |
|------|------|--------------|
| 检索命中率 | 相关文档是否被 Top-K 召回 | ≥ 90% |
| 答案准确率 | AI 答复是否基于召回片段正确生成 | ≥ 85% |
| 拒答率 | 置信度 < 0.6 的比例 | 合理范围 10-30% |
| 首轮解决率 | 一次对话就解决问题的比例 | ≥ 70% |

### 5.3 流式输出设计

- Qwen 大模型调用：同步等待全量返回 vs SSE 流式输出？
- 用户体验需要"正在检索/正在生成"的过渡态
- 流式输出下的敏感词后置校验如何实现？

## 六、数据与接口

### 6.1 核心数据模型未规划

建议至少提前定义以下核心表：

**对话会话表 (chat_session)**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | PK |
| session_id | varchar(64) | 会话唯一标识 |
| user_id | varchar(64) | 用户标识 |
| status | tinyint | 0进行中/1已关闭/2超时 |
| intent_type | varchar(32) | 识别出的意图类型 |
| confidence | decimal(3,2) | 意图识别置信度 |
| create_time | datetime | 创建时间 |

**消息记录表 (chat_message)**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | PK |
| session_id | varchar(64) | 关联会话 |
| role | varchar(16) | user/assistant/system |
| content | text | 消息内容 |
| tokens | int | Token消耗 |
| source_docs | json | 溯源文档列表 |
| confidence | decimal(3,2) | 答复置信度 |
| create_time | datetime | 创建时间 |

**工单表 (ticket)**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | PK |
| ticket_no | varchar(32) | 工单编号 |
| session_id | varchar(64) | 关联会话 |
| content | text | 工单内容 |
| status | tinyint | 0待处理/1处理中/2已办结 |
| create_time | datetime | 创建时间 |

**审计日志表 (audit_log)**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | PK |
| trace_id | varchar(64) | 全链路追踪ID |
| request_body | mediumtext | 请求体 |
| response_body | mediumtext | 响应体 |
| model_params | json | 模型调用参数 |
| retrieval_fragments | json | 检索召回片段 |
| latency_ms | int | 总耗时 |
| timestamp | datetime | 时间戳 |

### 6.2 缺少 API 接口规范

建议提前定义核心 RESTful 接口：

```
POST   /api/v1/chat/message          # 发送消息（支持 SSE 流式）
GET    /api/v1/chat/sessions         # 会话列表
GET    /api/v1/chat/sessions/{id}    # 会话历史
DELETE /api/v1/chat/sessions/{id}    # 删除会话
POST   /api/v1/tickets               # 创建工单
GET    /api/v1/tickets/{id}          # 查询工单状态
```

建议用 OpenAPI 3.0 / SpringDoc 先行定义，编码时靠注解自动生成文档和 SD K。

## 七、运维与观测

### 7.1 缺失项汇总

| 缺失项 | 建议方案 |
|--------|---------|
| 健康检查 | Spring Boot Actuator + `/health` + `/readiness` |
| 指标采集 | Micrometer → Prometheus：QPS、P95延迟、Token消耗、RAG检索耗时 |
| 日志规范 | 统一 JSON 格式 + TraceId 注入（SLF4J MDC），便于日志平台采集 |
| 告警规则 | 大模型调用失败率 > 5%、P95 > 5s、Qdrant 不可达、Redis 连接池耗尽 |
| 容器化 | 开发环境 Docker Compose 一键编排；生产环境根据客户环境适配（K8s / 裸机 systemd） |

### 7.2 全链路追踪

在日志拦截器中注入 TraceId，贯穿：`API网关 → 主Agent → 子Agent → RAG检索 → 模型调用 → 结果返回`，确保任一环节问题可快速定位。

## 八、技术栈风险提示

| 风险点 | 说明 | 建议 |
|--------|------|------|
| LangChain4j 1.14.0 | 版本较新，需确认与 Spring Boot 3.4.2 的兼容性 | 编码前做 POC 验证 |
| Qdrant 部署 | 需独立运维，团队是否有向量数据库运维经验 | 评估 Milvus / Elasticsearch 作为备选 |
| 百炼平台依赖 | Qwen3.7-Max 是云服务，私有化部署场景下网络可达性 | 确认部署环境能否访问公网/专线 |
| text-embedding-v4 | 也是百炼云服务，Embedding 生成依赖网络 | 评估本地化 Embedding 模型作为备选 |
