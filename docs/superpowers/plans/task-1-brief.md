# Task 1: 系统架构设计说明书

## Requirements

产出文件：`docs/design/系统架构设计说明书.md`

## 项目全局约束（必须遵循）

- 项目名：LAK-Agent（Legal Affairs Knowledge Agent Platform）
- 包名：com.lak.ai，服务名/数据库名：lak_ai_platform
- 服务端口：8080（业务），管理端口：8081
- 等保 2.0 基准（按三级预置扩展点）
- JDK17 + Spring Boot 3.4.2 + LangChain4j 1.14.0
- Qdrant（向量库）、Redis 7（会话+限流）、MySQL 8.0（业务+审计）
- 大模型：Qwen3.7-Max（百炼），向量模型：text-embedding-v4（百炼）
- 政务 6 个月日志留存、敏感词双向校验、所有 AI 答复必须溯源
- 私有化部署，所有组件无外部 SaaS 依赖（百炼 API 除外）

## 核心业务流程（来自总纲）

1. 统一前置拦截 → 主Agent意图识别+置信度判断 → 路由分发至对应子Agent → 业务处理 → 结果合规校验 → 统一结果封装返回 + 全链路日志落库
2. 政策咨询类：子AgentA → 政法政策库RAG检索 → 带原文溯源答复
3. 办事指引类：子AgentB → 办事指南库RAG检索 → 带原文溯源答复
4. 投诉建议类：子AgentC → 对话补齐工单必填项 → 调用内部工单接口创建工单 → 返回工单编号
5. 置信度<0.6 / 意图无法识别：统一兜底，返回人工客服联系方式，禁止AI自动作答

## 产出章节要求

### 1. 系统概述
- 项目背景与定位（1-2段）
- 系统目标（可用性/安全/可扩展）

### 2. 架构分层设计
每层必须包含：职责、核心组件、对外接口、技术选型

- **2.1 接入层**：API Gateway / Spring Interceptor 链，负责认证鉴权、限流、日志拦截、敏感词前置校验
- **2.2 业务编排层**：主Agent（意图识别+路由+置信度判断）+ 子Agent调度器，负责请求路由与Agent生命周期管理
- **2.3 能力层**：RAG引擎（向量检索+知识库查询）、对话管理器（会话状态机+上下文窗口）、工单适配器（工单接口封装）、合规校验器（敏感词后置校验+溯源校验）
- **2.4 数据层**：Qdrant（向量存储）、MySQL（业务数据+审计日志）、Redis（会话缓存+限流计数）、文件存储（MinIO/本地）

### 3. 组件拓扑与交互时序
使用 Mermaid 绘制 3 条核心链路时序图：
- **政策咨询链路**：用户提问 → 接入层前置拦截 → 主Agent意图识别 → 路由至子AgentA → RAG检索（Qdrant）→ 大模型生成 → 合规校验 → 溯源封装 → 返回
- **投诉建议链路**：用户提问 → 接入层前置拦截 → 主Agent意图识别 → 路由至子AgentC → 多轮信息补齐（Redis会话）→ 工单接口调用 → 返回工单编号
- **低置信度兜底链路**：用户提问 → 接入层前置拦截 → 主Agent意图识别（置信度<0.6）→ 直接返回人工客服引导（不调用大模型）

### 4. 部署架构
- **4.1 开发环境**：Docker Compose 编排全部组件，含服务依赖关系、端口映射、环境变量
- **4.2 生产环境参考**：裸机/VM 部署拓扑图（至少2节点），含 Nginx 反向代理、服务实例、中间件集群建议

### 5. 技术选型说明
对每个核心技术组件给出选型理由 + 备选方案：
- Spring Boot 3.4.2
- LangChain4j（vs Spring AI / 自研）
- Qdrant（vs Milvus / Elasticsearch）
- Qwen3.7-Max + text-embedding-v4（vs 开源本地模型）
- Redis 7 + MySQL 8.0

### 6. 安全架构集成
- 认证鉴权在架构中的位置
- 审计日志的采集与存储路径
- 敏感词双向校验的拦截点
- 通信加密（TLS）的端点

### 7. 扩展性设计
- 新增一个子Agent的步骤（最少几步、涉及哪些层）
- 切换大模型的最小变更范围
- 多租户架构预留（如果未来需要）

## 质量标准
- 每个架构决策都给出理由，不只是描述
- Mermaid 图表可直接渲染（GitHub/GitLab 原生支持）
- 端口、超时值、重试次数等配置给出具体建议值
- 章节之间逻辑自洽，无矛盾描述
