### 实体关系说明

| 主体 | 关系 | 客体 | 说明 |
|------|------|------|------|
| chat_session | 1:N | chat_message | 一个会话包含多条消息 |
| chat_session | 1:1 | ticket | 一个会话最多生成一个工单（投诉场景） |
| chat_session | 1:N | audit_log | 一个会话产生多条审计日志 |
| sys_user | 1:N | chat_session | 一个用户拥有多个会话 |
| sys_user | 1:N | audit_log | 一个用户操作产生多条审计日志 |
| sys_user | N:1 | sys_role | 一个用户属于一个角色 |
| sys_role | N:M | sys_permission | 一个角色拥有多个权限（通过关联表实现） |
| knowledge_document | 1:N | chat_message | 一个知识文档可以被多条消息引用溯源 |
| knowledge_document | 1:N | audit_log | 一个知识文档可以被多次检索并记录 |

## 2. 核心表 DDL（完整可执行）

### 全局约定

- **字符集**：`utf8mb4`，排序规则：`utf8mb4_unicode_ci`
- **存储引擎**：InnoDB
- **主键策略**：所有表使用 `bigint` 自增主键
- **时间字段**：所有业务表包含 `create_time` 和 `update_time`；审计日志表仅包含 `create_time`
- **命名规范**：表名小写蛇形，字段名小写蛇形，索引名前缀 `uk_`（唯一索引）/ `idx_`（普通索引）

### 2.1 chat_session（对话会话表）

会话生命周期摘要，记录每次对话的意图识别结果、状态流转和置信度。消息详细内容存储在 chat_message 表。

```sql
CREATE TABLE `chat_session` (
    `id`              BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `session_id`      VARCHAR(64)   NOT NULL                 COMMENT '会话唯一标识，UUID格式',
    `user_id`         BIGINT        NOT NULL                 COMMENT '关联用户ID',
    `status`          VARCHAR(32)   NOT NULL DEFAULT 'NEW'   COMMENT '会话状态: NEW/INTENT_CHECK/ANSWERING/COLLECT_INFO/FALLBACK/COMPLIANCE_CHECK/TICKET_SUBMIT/CLOSED',
    `intent_type`     VARCHAR(32)   DEFAULT NULL             COMMENT '意图类型: POLICY_CONSULT/PROCEDURE_GUIDE/COMPLAINT_SUGGEST/CHITCHAT/UNKNOWN',
    `confidence`      DECIMAL(3,2)  DEFAULT NULL             COMMENT '意图识别置信度(0.00-1.00)',
    `create_time`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_session_id` (`session_id`) USING BTREE COMMENT '会话ID唯一索引，支撑JOIN和直接查询',
    KEY `idx_user_id` (`user_id`) USING BTREE COMMENT '按用户查询会话列表',
    KEY `idx_create_time` (`create_time`) USING BTREE COMMENT '按时间范围查询和历史归档'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='对话会话表 — 记录会话生命周期与意图识别摘要';
```

### 2.2 chat_message（消息记录表）

每条消息的完整记录，包含用户输入、AI 答复、Token 消耗和溯源文档引用。

```sql
CREATE TABLE `chat_message` (
    `id`              BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `session_id`      VARCHAR(64)   NOT NULL                 COMMENT '关联会话ID，对应chat_session.session_id',
    `role`            VARCHAR(16)   NOT NULL                 COMMENT '消息角色: user/assistant/system',
    `content`         TEXT          NOT NULL                 COMMENT '消息内容（用户输入或AI答复）',
    `tokens`          INT           DEFAULT NULL             COMMENT 'Token消耗数（AI生成时记录，用户输入为NULL）',
    `source_docs`     JSON          DEFAULT NULL             COMMENT '溯源文档列表: [{"docId":"xxx","title":"政策文件","chunk":"原文片段","confidence":0.95}]',
    `confidence`      DECIMAL(3,2)  DEFAULT NULL             COMMENT '答复置信度',
    `create_time`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    PRIMARY KEY (`id`),
    KEY `idx_session_id` (`session_id`) USING BTREE COMMENT '按会话查询消息列表（对话展示）',
    KEY `idx_create_time` (`create_time`) USING BTREE COMMENT '按时间范围查询消息记录'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='消息记录表 — 完整对话消息内容与溯源引用';
```

**字段说明**：

- `role`：取值为 `user`（用户提问）、`assistant`（AI 答复）、`system`（系统提示/状态变更通知）
- `source_docs`：JSON 数组，存储 RAG 检索引用的知识文档溯源信息。结构见下方示例
- `tokens`：仅 assistant 角色消息记录 AI 答复的 Token 消耗，用于成本统计和监控

**source_docs JSON 结构示例**：

```json
[
  {
    "docId": "DOC-2026-00123",
    "title": "XX市行政诉讼程序规定（2026版）",
    "chunk": "第三十五条 当事人对一审判决不服的，可于收到判决书之日起十五日内...",
    "confidence": 0.92,
    "page": 5
  },
  {
    "docId": "DOC-2026-00124",
    "title": "XX市行政执法工作指引",
    "chunk": "2.3 行政处罚决定的作出应当经过法制审核...",
    "confidence": 0.87,
    "page": 12
  }
]
```

### 2.3 ticket（工单表）

投诉建议场景下，AgentC（投诉建议Agent）完成信息采集后向内部工单系统提交的工单记录。1:1 关联 `chat_session`。

```sql
CREATE TABLE `ticket` (
    `id`                  BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `ticket_no`           VARCHAR(32)   NOT NULL                 COMMENT '工单编号，格式建议:TK-yyyyMMdd-xxxx',
    `session_id`          VARCHAR(64)   NOT NULL                 COMMENT '关联会话ID，1:1关系',
    `complaint_type`      VARCHAR(32)   DEFAULT NULL             COMMENT '投诉类型: LAW_ENFORCEMENT/SERVICE_COMPLAINT/DISCIPLINE_VIOLATION/OTHER',
    `contact_name`        VARCHAR(64)   DEFAULT NULL             COMMENT '联系人姓名',
    `contact_phone`       VARCHAR(20)   DEFAULT NULL             COMMENT '联系电话（等保要求：存储加密）',
    `description`         TEXT          DEFAULT NULL             COMMENT '问题描述',
    `attachment_url`      VARCHAR(512)  DEFAULT NULL             COMMENT '附件URL（MinIO路径或外部链接）',
    `status`              VARCHAR(32)   NOT NULL DEFAULT 'PENDING' COMMENT '工单状态: PENDING/PROCESSING/COMPLETED/FAILED',
    `external_ticket_id`  VARCHAR(64)   DEFAULT NULL             COMMENT '外部工单系统返回的工单ID',
    `create_time`         DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`         DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_ticket_no` (`ticket_no`) USING BTREE COMMENT '工单编号唯一索引',
    UNIQUE KEY `uk_session_id` (`session_id`) USING BTREE COMMENT '与会话1:1关联',
    KEY `idx_status` (`status`) USING BTREE COMMENT '按状态查询工单列表（客服工作台）',
    KEY `idx_create_time` (`create_time`) USING BTREE COMMENT '按时间范围查询工单'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='工单表 — 投诉建议场景生成的工单数据';
```

**字段说明**：

- `contact_phone`：等保 2.0 要求敏感字段加密存储。应用层使用 AES-256 加密后写入，读取时由应用层解密（不依赖 MySQL 内置加密函数，避免影响索引功能）
- `session_id` 设置了唯一索引（而非普通索引），以保证与 `chat_session` 的 1:1 关系在数据库层面得到强制约束
- `external_ticket_id`：当工单适配器成功调用内部工单系统后，记录外部系统的返回 ID，用于后续状态查询

### 2.4 audit_log（审计日志表 — 按月分表）

全链路审计日志，记录请求到响应的完整数据流，包含意图识别、RAG 检索、大模型调用、合规校验等全环节信息。**等保 2.0 特性**：该表仅允许 INSERT + SELECT，禁止 UPDATE/DELETE。

**表名规则**：`audit_log_yyyyMM`，如 `audit_log_202606`（当前月份示例）

```sql
CREATE TABLE `audit_log_202606` (
    `id`                    BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `trace_id`              VARCHAR(64)   NOT NULL                 COMMENT '全链路追踪ID，接入层TraceIdFilter生成',
    `session_id`            VARCHAR(64)   DEFAULT NULL             COMMENT '关联会话ID',
    `user_id`               BIGINT        DEFAULT NULL             COMMENT '操作用户ID',
    `request_body`          MEDIUMTEXT    DEFAULT NULL             COMMENT '请求体（用户原始输入）',
    `response_body`         MEDIUMTEXT    DEFAULT NULL             COMMENT '响应体（AI最终答复）',
    `intent_type`           VARCHAR(32)   DEFAULT NULL             COMMENT '意图类型: POLICY_CONSULT/PROCEDURE_GUIDE/COMPLAINT_SUGGEST/CHITCHAT/UNKNOWN',
    `confidence`            DECIMAL(3,2)  DEFAULT NULL             COMMENT '意图识别置信度',
    `model_params`          JSON          DEFAULT NULL             COMMENT '大模型调用参数: {model, temperature, maxTokens, systemPrompt, messages}',
    `model_response`        MEDIUMTEXT    DEFAULT NULL             COMMENT '大模型原始返回（完整JSON）',
    `retrieval_fragments`   JSON          DEFAULT NULL             COMMENT 'RAG检索召回片段: [{collection, docId, chunk, score, rank}]',
    `latency_ms`            INT           DEFAULT NULL             COMMENT '全链路总耗时(毫秒)',
    `status`                VARCHAR(16)   DEFAULT NULL             COMMENT '处理结果: SUCCESS/FAIL/FALLBACK',
    `error_message`         TEXT          DEFAULT NULL             COMMENT '异常信息（status=FAIL时填写）',
    `create_time`           DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录时间（分区键）',
    PRIMARY KEY (`id`),
    KEY `idx_trace_id` (`trace_id`) USING BTREE COMMENT 'TraceID索引，支撑全链路追踪查询',
    KEY `idx_session_id` (`session_id`) USING BTREE COMMENT '按会话查询审计轨迹',
    KEY `idx_create_time` (`create_time`) USING BTREE COMMENT '按时间范围查询审计记录',
    KEY `idx_status` (`status`) USING BTREE COMMENT '按结果状态查询异常记录'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='审计日志表(202606) — 全链路请求响应与模型调用日志，仅INSERT+SELECT';
```

**字段说明**：

- `trace_id`：由接入层的 TraceIdFilter 在请求入口处通过 UUID 或 Snowflake 算法生成，注入 SLF4J MDC，全链路日志通过此 ID 关联
- `request_body` / `response_body`：使用 MEDIUMTEXT 而非 TEXT（最大 16MB），因 Policy/Guide 场景下涉诉材料可能包含较长文本
- `model_params`：JSON 结构，完整记录每次大模型调用的参数，用于后续审计回溯和问题排查
- `model_response`：MEDIUMTEXT 存储大模型原始返回 JSON，便于事后分析模型行为。部分模型返回可能超过 64KB，因此选择 MEDIUMTEXT（最大 16MB）
- `retrieval_fragments`：JSON 结构，记录 RAG 检索返回的 Top-K 片段及其得分

**retrieval_fragments JSON 结构示例**：

```json
[
  {
    "collection": "lak_policy_docs",
    "docId": "DOC-2026-00123",
    "title": "XX市行政诉讼程序规定（2026版）",
    "chunk": "第三十五条...",
    "score": 0.92,
    "rank": 1
  },
  {
    "collection": "lak_policy_docs",
    "docId": "DOC-2026-00124",
    "title": "XX市行政执法工作指引",
    "chunk": "2.3 行政处罚...",
    "score": 0.87,
    "rank": 2
  }
]
```

**model_params JSON 结构示例**：

```json
{
  "model": "qwen3.7-max",
  "temperature": 0.3,
  "maxTokens": 2048,
  "topP": 0.9,
  "systemPrompt": "你是一名专业的政法领域AI助手...",
  "messages": [
    {"role": "user", "content": "请问行政诉讼的时效是多久？"}
  ],
  "tools": [],
  "provider": "dashscope"
}
```

### 2.5 knowledge_document（知识文档元数据 — P1 预留）

知识库中文档的元数据信息，对应的向量数据存储在 Qdrant 中。该表为 P1 阶段使用，当前阶段（P0）仅定义表结构，不实现 CRUD 接口。

```sql
CREATE TABLE `knowledge_document` (
    `id`                BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `doc_id`            VARCHAR(64)   NOT NULL                 COMMENT '文档唯一标识，格式:DOC-yyyyMMdd-xxxx',
    `title`             VARCHAR(256)  NOT NULL                 COMMENT '文档标题',
    `doc_type`          VARCHAR(32)   NOT NULL                 COMMENT '文档类型: POLICY/PROCEDURE/TEMPLATE',
    `file_url`          VARCHAR(512)  DEFAULT NULL             COMMENT '文件存储路径（MinIO或本地路径）',
    `effective_date`    DATE          DEFAULT NULL             COMMENT '生效日期',
    `expire_date`       DATE          DEFAULT NULL             COMMENT '废止日期（NULL=长期有效）',
    `status`            VARCHAR(16)   NOT NULL DEFAULT 'ACTIVE' COMMENT '文档状态: ACTIVE/EXPIRED/DRAFT',
    `chunk_count`       INT           DEFAULT 0                COMMENT '分段数量（对应Qdrant中的向量数）',
    `qdrant_collection` VARCHAR(64)   DEFAULT NULL             COMMENT '对应Qdrant Collection: lak_policy_docs/lak_procedure_docs',
    `create_time`       DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`       DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_doc_id` (`doc_id`) USING BTREE COMMENT '文档唯一标识索引',
    KEY `idx_doc_type` (`doc_type`) USING BTREE COMMENT '按文档类型查询（知识管理列表）',
    KEY `idx_status` (`status`) USING BTREE COMMENT '按文档状态过滤（仅查询有效文档）',
    KEY `idx_effective_date` (`effective_date`) USING BTREE COMMENT '按生效日期查询（到期提醒）'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='知识文档元数据表 — 知识库文档管理与检索（P1预留）';
```

### 2.6 sys_user（用户表 — RBAC 预留）

系统用户表，采用 RBAC 模型实现权限管理。密码需使用 BCrypt 加密存储。

```sql
CREATE TABLE `sys_user` (
    `id`              BIGINT        NOT NULL AUTO_INCREMENT  COMMENT '主键ID',
    `username`        VARCHAR(64)   NOT NULL                 COMMENT '用户名（登录名）',
    `password_hash`   VARCHAR(256)  NOT NULL                 COMMENT '密码哈希（BCrypt加密）',
    `real_name`       VARCHAR(64)   DEFAULT NULL             COMMENT '真实姓名',
    `email`           VARCHAR(128)  DEFAULT NULL             COMMENT '电子邮箱',
    `phone`           VARCHAR(20)   DEFAULT NULL             COMMENT '手机号（等保要求：存储加密）',
    `status`          VARCHAR(16)   NOT NULL DEFAULT 'ACTIVE' COMMENT '用户状态: ACTIVE/DISABLED/LOCKED',
    `role_id`         BIGINT        DEFAULT NULL             COMMENT '关联角色ID',
    `create_time`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `update_time`     DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_username` (`username`) USING BTREE COMMENT '用户名唯一索引',
    KEY `idx_role_id` (`role_id`) USING BTREE COMMENT '按角色查询用户列表',
    KEY `idx_status` (`status`) USING BTREE COMMENT '按用户状态过滤'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='系统用户表 — RBAC用户管理';
```

**安全说明**：

- `password_hash`：使用 BCrypt 加密存储，加盐由算法内置。禁止明文存储或使用 MD5/SHA1 等弱哈希算法
- `phone`：等保 2.0 要求敏感字段加密存储，建议采用 AES-256 应用层加密（与 ticket.contact_phone 一致）

### 2.9 sys_role_permission（角色权限关联表 — RBAC 预留）

```sql
CREATE TABLE `sys_role_permission` (
    `role_id`       BIGINT        NOT NULL  COMMENT '角色ID',
    `permission_id` BIGINT        NOT NULL  COMMENT '权限ID',
```

## 3. 索引策略

### 3.1 索引设计原则

1. **覆盖查询场景**：每个索引的设计目标为解决一个或多个可预期的查询模式
2. **联合索引优先**：当查询条件包含多个字段时，优先设计联合索引，避免数据库使用 Index Merge 优化（可能导致性能不稳定）
3. **区分度优先**：区分度高的列放在联合索引左侧
4. **避免冗余索引**：已有索引 A(a, b) 时，不再建单列索引 idx_a（MySQL 可利用左前缀原则）

### 3.2 各表索引详解

#### chat_session 索引

| 索引名 | 类型 | 字段 | 覆盖查询场景 | 设计理由 |
|--------|------|------|-------------|---------|
| `uk_session_id` | UNIQUE BTREE | `session_id` | 1. 按 sessionId 查询会话详情<br>2. chat_message/ticket/audit_log 的 JOIN 依赖 | session_id 是 VARCHAR(64) 的业务主键，在所有关联表中作为外键使用，必须建立唯一索引 |
| `idx_user_id` | BTREE | `user_id` | 1. 按用户查询历史会话列表<br>2. 管理后台按用户维度统计 | 查询模式为 `WHERE user_id = ? ORDER BY create_time DESC`，若性能不足可改为联合索引 `(user_id, create_time)` |
| `idx_create_time` | BTREE | `create_time` | 1. 按时间范围查询会话（管理后台）<br>2. 定时任务扫描超时会话 | 会话的归档和清理依赖时间范围扫描 |

#### chat_message 索引

| 索引名 | 类型 | 字段 | 覆盖查询场景 | 设计理由 |
|--------|------|------|-------------|---------|
| `idx_session_id` | BTREE | `session_id` | 1. 按会话查询全部消息（对话展示）<br>2. chat_session JOIN 查询 | 会话详情页读取消息列表是该表最频繁的查询，必须建立索引。区分度很高（每个 session_id 对应 10-30 条消息），无需联合索引 |
| `idx_create_time` | BTREE | `create_time` | 1. 按时间范围查询消息记录<br>2. 审计与数据归档 | 归档任务扫描旧消息时使用 |

**关于 source_docs 的索引**：MySQL 8.0 支持 JSON 字段的多值索引（Multi-Value Index），可在 `source_docs->>"$.docId"` 上建立索引以支持按溯源文档检索消息。但考虑到 P0 阶段此查询场景低频（属于审计事后分析），暂不建立 JSON 索引，保留为 P1 优化项。

#### ticket 索引

| 索引名 | 类型 | 字段 | 覆盖查询场景 | 设计理由 |
|--------|------|------|-------------|---------|
| `uk_ticket_no` | UNIQUE BTREE | `ticket_no` | 1. 按工单编号精确查询<br>2. 与外部系统交互时通过 ticket_no 关联 | 工单编号是面向用户/外部系统的业务标识，必须唯一 |
| `uk_session_id` | UNIQUE BTREE | `session_id` | 1. 按会话查询关联的工单<br>2. 保证 1:1 关系完整性 | 使用 UNIQUE 而非普通 KEY，所以能在数据库层面强制一个会话最多对应一个工单 |
| `idx_status` | BTREE | `status` | 1. 客服工作台按状态筛选工单<br>2. 后台统计各状态工单数量 | 工单状态是高频筛选条件，区分度中等（4 个枚举值），可有效过滤数据 |
| `idx_create_time` | BTREE | `create_time` | 1. 工单列表的时间排序<br>2. 归档和统计 | 工单列表默认按时间倒序展示 |

**关于 contact_phone 索引**：由于 phone 字段加密存储，无法在数据库层面执行 `=` 查询和建立索引。若后续出现"按手机号查询工单"的业务需求，建议通过 phone 的 SHA-256 哈希值建立侧索引（单独存储 hash 列用于等值查询，原始值仍保持加密），此为 P1 优化项。

#### audit_log 索引策略（分区表索引）

审计日志表为按月分表（`audit_log_yyyyMM`），每个分表独立建立索引。

| 索引名 | 类型 | 字段 | 覆盖查询场景 | 设计理由 |
|--------|------|------|-------------|---------|
| `idx_trace_id` | BTREE | `trace_id` | 1. 全链路追踪——按 traceId 查询一次请求的全部环节 | 全链路追踪是审计日志最核心的查询场景。trace_id 在单表中区分度极高（UUID 级别），索引效率最高 |
| `idx_session_id` | BTREE | `session_id` | 1. 按会话查询该会话的所有审计记录 | 排查具体会话的问题时需要查看会话的完整审计轨迹 |
| `idx_create_time` | BTREE | `create_time` | 1. 按时间范围查询审计日志<br>2. 归档任务确定需要归档的数据 | create_time 是分表键同时也是查询条件。当查询限定在单个分表内时，该索引可高效进行范围扫描 |
| `idx_status` | BTREE | `status` | 1. 查询异常记录（status=FAIL）<br>2. 监控告警——发现连续异常 | 运维监控需要快速识别异常记录，status 的区分度较好（3 个值），可以快速定位问题时段 |

**分区表索引特殊说明**：

1. **分表即分区**：按月分表策略下（`audit_log_202606` 等表名），每个分表是一个独立的物理表。查询时应用层通过 Mybatis-Plus 动态表名拦截器路由到正确的分表，因此索引是分表级别的，非 MySQL 内置 PARTITION
2. **避免跨表查询**：审计日志查询必须携带时间范围，将查询限定在最多 2-3 个分表中。禁止无时间范围的全局扫描
3. **复合索引暂不建立**：考虑到单表数据量（百万级/月）和 MySQL 8.0 的索引性能，当前阶段单列索引已足够覆盖所有查询场景。若后续发现 `(session_id, create_time)` 的复合查询成为瓶颈，可在分表上追加复合索引

#### knowledge_document 索引

| 索引名 | 类型 | 字段 | 覆盖查询场景 | 设计理由 |
|--------|------|------|-------------|---------|
| `uk_doc_id` | UNIQUE BTREE | `doc_id` | 1. 按文档标识精确查询<br>2. 知识库同步时对比 | doc_id 是文档的业务标识，必须唯一 |
| `idx_doc_type` | BTREE | `doc_type` | 1. 按文档类型筛选（政策/指引/模板）<br>2. 知识管理后台的分类展示 | 区分度好（3 个类型值），可有效过滤 |
| `idx_status` | BTREE | `status` | 1. 仅查询有效文档（ACTIVE）<br>2. 知识库检索前的状态过滤 | RAG 引擎查询知识库时只检索 status=ACTIVE 的文档 |
| `idx_effective_date` | BTREE | `effective_date` | 1. 查询即将生效/已生效的文档<br>2. 过期文档提醒（cron job） | 辅助定期扫描，处理文档生命周期 |

### 3.3 索引维护注意事项

- **避免在线 DDL 锁表**：MySQL 8.0 支持 `ALGORITHM=INPLACE, LOCK=NONE` 的在线 DDL，生产环境新增索引时使用：`ALTER TABLE xxx ADD INDEX ... ALGORITHM=INPLACE, LOCK=NONE`
- **定期分析慢查询**：通过 `slow_query_log` 监控索引使用情况，对全表扫描的查询及时补充索引
- **审计日志分表索引同步**：每月创建新分表时，DDL 必须包含完整索引定义。建议通过 Mybatis-Plus 拦截器自动执行建表 SQL，或通过 `init.sql` 脚本统一管理

## 4. 分表与归档策略

### 4.1 audit_log 按月分表方案

#### 4.1.1 分表规则

- **表名格式**：`audit_log_yyyyMM`（如 `audit_log_202606`）
- **创建时机**：每月 1 日凌晨 00:30，由定时任务自动创建下月分表
- **DDL 来源**：预定义 DDL 模板，仅替换表名后缀。见 `sql/init.sql` 中的完整 CREATE TABLE 语句
- **路由策略**：应用层通过 Mybatis-Plus 动态表名拦截器（DynamicTableNameInnerInterceptor）实现

#### 4.1.2 Mybatis-Plus 动态表名拦截器实现

```java
// 动态表名拦截器配置
@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        DynamicTableNameInnerInterceptor dynamicTableName = new DynamicTableNameInnerInterceptor();
        dynamicTableName.setTableNameHandler((sql, tableName) -> {
            if ("audit_log".equals(tableName)) {
                // 从 ThreadLocal 获取当前月份的表名后缀（由 AuditContextHolder 在请求入口注入）
                String monthSuffix = AuditContextHolder.getMonthSuffix();
                return tableName + "_" + monthSuffix;
            }
            return tableName;
        });
        interceptor.addInnerInterceptor(dynamicTableName);
        return interceptor;
    }
}
```

**关键设计说明**：

1. `AuditContextHolder` 的 `monthSuffix` 值在请求入口由 `AuditLogInterceptor` 根据 `create_time` 确定
2. 定时任务（如归档）查询时，由调用方在 ThreadLocal 中设置目标月份，实现任意月份分表的读写
3. 若遇到跨月请求（极低概率），以请求到达时间所在月份作为分表路由依据

#### 4.1.3 分表创建定时任务

```sql
-- 每月1日 00:30 执行的分表创建存储过程（供定时任务调用）
-- 使用预编译语句方式，避免 SQL 注入风险
SET @nextMonth = DATE_FORMAT(DATE_ADD(LAST_DAY(CURDATE()), INTERVAL 1 DAY), '%Y%m');
SET @sql = CONCAT('
```

### 4.2 6 个月数据归档方案

#### 4.2.1 归档策略

| 时间区间 | 存储位置 | 查询方式 | 说明 |
|---------|---------|---------|------|
| 当前月至 5 个月前 | MySQL 分表 | 应用层查询（正常访问） | 在线数据，保持 6 个月在线 |
| 6 个月之前 | 归档文件（CSV/Parquet） | 离线查询，后台导入工具 | 物理删除或移动到归档表 |

#### 4.2.2 归档定时任务

**执行频率**：每月 1 日凌晨 02:00（避开业务高峰期）

**归档流程**：

```
1. 计算归档目标月份：当前月 - 7（保留6个月在线）
2. 检查目标分表存在且可读
3. 导出数据至归档文件（CSV 格式，UTF-8 编码）
4. 验证导出数据的完整性（行数对比 + MD5 校验）
5. 执行 DROP TABLE 删除已归档的分表
6. 记录归档日志（写入归档元数据表或记录文件）
```

**归档 SQL**（应用层执行，通过定时任务框架调度）：

```sql
-- 步骤 1: 导出数据（应用层通过 SELECT INTO OUTFILE 或 Mybatis-Plus 流式查询）
-- MySQL 8.0 原生导出方式（需要 FILE 权限）：
SELECT *
FROM audit_log_202512
INTO OUTFILE '/data/archive/audit_log_202512.csv'
FIELDS TERMINATED BY ',' OPTIONALLY ENCLOSED BY '"'
LINES TERMINATED BY '\n';

-- 步骤 2: 验证通过后删除已归档分表
DROP TABLE IF EXISTS `audit_log_202512`;
```

#### 4.2.3 归档数据的查询策略

| 查询场景 | 策略 | 实现方式 |
|---------|------|---------|
| 最近 6 个月的审计日志 | 直接查询 MySQL 分表 | Mybatis-Plus 动态表名 |
| 6 个月前的审计日志（回溯查询） | 提示用户联系管理员导入归档 | 前端 UI 提示 + 后台归档导入工具 |
| 合规检查/审计调查 | 管理员通过归档导入工具将目标分表恢复到 MySQL | `LOAD DATA INFILE` 恢复到临时表 |
| 法定数据保留 | 归档文件至少保留 2 年（以归档文件形式存储） | 文件系统 + 冷存储 |
