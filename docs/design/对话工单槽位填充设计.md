# 对话工单槽位填充设计

> 通过多轮对话创建工单，支持打断处理、冗余输入清洗、槽位修改。

## 1. 概述

当前 ComplaintAgent 只返回静态文本引导用户去 `/tickets` 页面填写表单。改造目标：用户在对话中自然完成工单创建，系统理解用户意图、提取关键信息、处理各种干扰。

### 核心场景

| 场景 | 用户输入 | 系统行为 |
|------|---------|---------|
| 正常填写 | "治安投诉" | 写入槽位，推进下一问题 |
| 冗余输入 | "我张三" | 清洗为"张三"后写入 |
| 打断 | "殴打他人违法么" | 挂起槽位，路由 PolicyAgent 回答，附带回钩 |
| 修改 | "改一下手机号" | 回退到手机号槽位，重新询问 |
| 取消 | "算了不填了" | 退出 COLLECT_INFO，关闭会话 |

## 2. 架构

### 2.1 路由改造

```
ChatService.processMessage()
├─ getStatus() == COLLECT_INFO ?
│   ├─ YES → SlotFillingEngine.processResponse()（跳过意图分类）
│   └─ NO  → 正常意图分类，COMPLAINT_SUGGEST 时进入 COLLECT_INFO
```

### 2.2 槽位处理管道

```
用户输入（已在 COLLECT_INFO 状态）
    │
    ├─ Step 1: 规则预清洗
    │   ├─ 去口头前缀：/^[我我们]?是?叫?/
    │   ├─ 手机号正则直提：/(1[3-9]\d{9})/
    │   └─ 退出关键词："算了"/"不填了"/"取消" → action=cancel
    │
    ├─ Step 2: LLM 结构化抽取（~400 tokens / 次）
    │   输出 JSON: {action, targetSlot, extractedValue, confidence, reasoning}
    │
    ├─ Step 3: JSON Schema 校验
    │   格式错/字段缺失/值异常 → reprompt × 2
    │
    ├─ Step 4: 按 action 分发
    │   ├─ fill       → 写槽值，推进下一槽位
    │   ├─ modify     → 回退到指定槽位
    │   ├─ chitchat   → 挂起，回答闲聊，附回钩
    │   ├─ new_intent → 挂起，路由对应 Agent，记录打断轮次
    │   └─ cancel     → 退出，CLOSED
    │
    └─ Step 5: 完成检测
        所有必填槽位已填 AND（可选槽位已填 OR 用户说"提交" OR 轮次≥5）
        → TicketAdapter.createTicket()
```

### 2.3 打断处理：挂起-恢复

```
打断触发 → Redis 写入 interruption:{active:true, rounds:1, resumeSlot:N}
    → 路由到对应 Agent 处理
    → 答案末尾附："（还有其他疑问请继续，否则我将回到工单填写）"
    → rounds < 3 时继续处理追问
    → rounds ≥ 3 强制回归
    → 用户30s无追问自动回归
```

### 2.4 自然语言摘要（NL-DST）

每轮槽值更新后维护一句自然语言摘要，作为 LLM Prompt 的上下文：

> *已收集：投诉类型为治安投诉，联系人为张三，手机号138****8000。还需填写：问题描述、附件（可选）。*

相比键值对，LLM 对自然语言摘要的理解准确率更高。

## 3. Redis 数据结构

在 session Hash 中新增字段：

| Key | 类型 | 说明 |
|-----|------|------|
| `slotValues` | JSON Map | 已有的槽位键值对 |
| `currentSlot` | int | 当前槽位索引 |
| `fillRound` | int | 已填充轮次 |
| `nlSummary` | String | 自然语言摘要 |
| `interruption` | JSON | `{active, rounds, resumeSlot}` |

## 4. 槽位定义

| 槽位 | 必填 | 验证 |
|------|------|------|
| complaintType | 是 | 枚举：治安投诉/窗口服务投诉/派出所民警投诉/其他 |
| contactName | 是 | 长度 1-64 |
| contactPhone | 是 | 正则 `^1[3-9]\d{9}$` |
| description | 是 | 长度 10-2000 |
| attachment | 否 | URL 或 "无" |

## 5. LLM Prompt 模板

```
你是工单信息提取助手。当前收集槽位：{slotLabel}（{slotName}）。
已收集信息摘要：{nlSummary}
用户消息："{userMessage}"

严格输出JSON（不要额外文字）：
{
  "action": "fill|modify|chitchat|new_intent|cancel",
  "targetSlot": "{slotName}",
  "extractedValue": "提取的值",
  "confidence": 0.95,
  "reasoning": "简短判断"
}

规则：
- 口头前缀（"我叫""我是""我"）提取时去掉
- 手机号必须11位1开头
- 修改意图用 action="modify"
- 闲聊/问候用 action="chitchat"
- 政策法律问题用 action="new_intent"
- 取消放弃用 action="cancel"
```

## 7. 未来优化：模型蒸馏方案

> 参考论文：COLING 2025 [Zero-shot Slot Filling in the Age of LLMs](https://arxiv.org/abs/2411.18980)、IBM Research [Industry-Grade Zero-Shot Slot-Filling](https://arxiv.org/abs/2406.08848)

### 7.1 目标

将槽位提取从大模型（Qwen-Max）迁移到小模型（Qwen-0.5B 或 Llama-3-8B），保持 90%+ 的提取准确率同时将延迟从 ~1s 降至 ~50ms。

### 7.2 实施步骤

1. **积累训练数据**：将所有用户输入 + LLM 提取结果脱敏存储，目标 5000-10000 条
2. **知识蒸馏**：大模型作为 Teacher 标注，小模型作为 Student 微调（LoRA）
3. **分层推理**：小模型首轮推理（50ms），confidence < 0.8 时大模型兜底
4. **持续学习**：兜底样本自动加入训练集，定期重训

### 7.3 蒸馏技术要点

- Teacher: Qwen3.7-Max（或 Qwen-Plus），输出结构化 JSON + reasoning
- Student: Qwen-0.5B / Llama-3-8B，仅输出 JSON，不要 reasoning
- 损失函数：KL 散度（Teacher 输出分布 vs Student 输出分布）
- 部署：vLLM 或 Ollama 本地推理，50-100ms 首 Token
