## 2. 统一响应规范

### 2.1 成功响应格式

所有成功接口统一返回以下 JSON 结构：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| code | Integer | 是 | HTTP 状态码（与 HTTP Status 一致） |
| message | String | 是 | 提示信息，成功时为 `"success"` |
| data | Object | 是 | 业务数据载体，接口不同结构不同 |
| traceId | String | 是 | 全链路追踪 ID，UUID 格式 |

**示例**（登录成功）：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "dGhpcyBpcyBhIHJlZnJlc2g...",
    "expiresIn": 7200,
    "userInfo": {
      "userId": 1,
      "username": "admin",
      "realName": "系统管理员",
      "roleCode": "ADMIN"
    }
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### 2.2 分页响应格式

分页查询接口使用统一的分页响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [ ... ],
    "total": 100,
    "page": 1,
    "size": 20
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data.records | Array | 是 | 当前页数据列表 |
| data.total | Long | 是 | 总记录数 |
| data.page | Integer | 是 | 当前页码，从 1 开始 |
| data.size | Integer | 是 | 每页条数 |

### 2.3 错误响应格式

```json
{
  "code": 401,
  "message": "用户名或密码错误",
  "errorCode": "LAK-01-002",
  "data": null,
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| code | Integer | 是 | HTTP 状态码 |
| message | String | 是 | 面向用户的可读错误消息（不应包含敏感信息） |
| errorCode | String | 是 | 系统内部错误码，格式 `LAK-XX-XXX` |
| data | Object | 否 | 错误详情（参数校验错误时返回具体字段） |
| traceId | String | 是 | 全链路追踪 ID |

### 2.4 traceId 说明

- **生成机制**：在请求入口由 `TraceIdFilter` 过滤器中通过 UUID 或 Snowflake 算法生成
- **传递方式**：注入 SLF4J MDC，全链路日志通过此 ID 关联
- **HTTP Header**：服务间调用在请求头中传递 `X-Trace-Id`
- **响应体**：所有 API 响应均包含 `traceId` 字段，客户端可将此 ID 提供给技术支持用于问题排查
- **审计日志**：`traceId` 作为 audit_log 表的核心索引字段，支撑全链路追踪

## 3. 认证接口详细设计

### 3.1 POST /api/v1/auth/login

#### 3.1.1 接口说明

用户登录接口，接收用户名密码和验证码，认证成功后返回 JWT Token 对。**无需 JWT 认证**。

#### 3.1.2 Request Body

```json
{
  "username": "admin",
  "password": "admin123456",
  "captchaKey": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "captchaCode": "A3B7"
}
```

#### 3.1.3 Request Schema

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|---------|------|
| username | String | 是 | 2-64 字符，字母数字下划线 | 登录用户名 |
| password | String | 是 | 8-64 字符，至少包含字母和数字 | 登录密码（明文传输，服务端使用 BCrypt 验证） |
| captchaKey | String | 是 | UUID 格式 | 验证码 Key，通过 `GET /api/v1/auth/captcha` 获取 |
| captchaCode | String | 是 | 4 位字母数字 | 验证码结果 |

#### 3.1.4 Response Body（成功）

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "def50200a5b7c8d9...",
    "expiresIn": 7200,
    "userInfo": {
      "userId": 1,
      "username": "admin",
      "realName": "系统管理员",
      "roleCode": "ADMIN"
    }
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### 3.1.5 Response Schema

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data.accessToken | String | 是 | JWT Access Token，有效期 2h，用于后续请求认证 |
| data.refreshToken | String | 是 | Refresh Token，有效期 7d，用于刷新 Access Token |
| data.expiresIn | Integer | 是 | Access Token 过期时间（秒），7200 |
| data.userInfo | Object | 是 | 用户基本信息 |
| data.userInfo.userId | Long | 是 | 用户 ID，对应 `sys_user.id` |
| data.userInfo.username | String | 是 | 用户名 |
| data.userInfo.realName | String | 否 | 真实姓名 |
| data.userInfo.roleCode | String | 否 | 角色编码，如 `ADMIN`、`OPERATOR`、`AUDITOR` |

#### 3.1.6 错误场景

| 触发条件 | HTTP 状态码 | errorCode | message |
|---------|------------|-----------|---------|
| 验证码错误 | 400 | LAK-01-004 | 验证码错误 |
| 验证码已过期 | 400 | LAK-01-005 | 验证码已过期，请重新获取 |
| 用户名不存在 | 401 | LAK-01-001 | 用户名或密码错误 |
| 密码错误 | 401 | LAK-01-002 | 用户名或密码错误 |
| 账户已禁用 | 403 | LAK-01-003 | 账户已被禁用，请联系管理员 |
| 参数校验失败 | 400 | LAK-98-601 | 参数校验失败（data 字段返回具体字段错误） |

> **安全说明**：用户名不存在和密码错误使用相同消息 "用户名或密码错误"，避免枚举用户名。

**参数校验失败示例**：

```json
{
  "code": 400,
  "message": "参数校验失败",
  "errorCode": "LAK-98-601",
  "data": {
    "username": "用户名长度必须在 2-64 个字符之间",
    "password": "密码长度必须在 8-64 个字符之间"
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### 3.2 GET /api/v1/auth/captcha

#### 3.2.1 接口说明

获取登录验证码，返回验证码 Key 和 Base64 编码的图片。**无需 JWT 认证**。验证码有效期 5 分钟。

#### 3.2.2 Response Body

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "captchaKey": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "captchaImage": "data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAASw..."
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### 3.2.3 Response Schema

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data.captchaKey | String | 是 | 验证码唯一标识（UUID），登录时提交此 Key |
| data.captchaImage | String | 是 | Base64 编码的 PNG 图片，含 `data:image/png;base64,` 前缀 |

#### 3.2.4 后端处理说明

- 验证码使用基于 Redis 的缓存机制：`captcha:{captchaKey}` → `{code}`
- 验证码生成逻辑：4 位随机字母数字（排除易混淆字符如 0/O、1/l/I）
- 缓存 TTL：5 分钟（300 秒）
- 验证码图片使用开源库生成（如 EasyCaptcha），支持中文干扰线

#### 3.2.5 无错误场景

该接口为纯获取接口，成功始终返回 200。仅在系统层面错误时返回 500。

### 3.3 POST /api/v1/auth/refresh

#### 3.3.1 接口说明

使用 Refresh Token 获取新的 Access Token。**无需 JWT 认证**（此时 Access Token 可能已过期）。

#### 3.3.2 Request Body

```json
{
  "refreshToken": "def50200a5b7c8d9..."
}
```

#### 3.3.3 Request Schema

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|---------|------|
| refreshToken | String | 是 | 非空，长度 32-512 | 登录时获取的 Refresh Token |

#### 3.3.4 Response Body

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "ghij67890klmn12345...",
    "expiresIn": 7200
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### 3.3.5 Response Schema

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data.accessToken | String | 是 | 新的 Access Token |
| data.refreshToken | String | 是 | 新的 Refresh Token（每次刷新都会轮换） |
| data.expiresIn | Integer | 是 | Access Token 过期时间（秒），7200 |

> **安全说明**：采用 Refresh Token 轮换机制（Rotation），每次刷新时旧的 Refresh Token 立即失效，返回新的 Refresh Token。防止 Refresh Token 泄露后的长期滥用。

#### 3.3.6 错误场景

| 触发条件 | HTTP 状态码 | errorCode | message |
|---------|------------|-----------|---------|
| Refresh Token 无效 | 401 | LAK-01-006 | 无效的 Refresh Token |
| Refresh Token 已过期 | 401 | LAK-01-007 | Refresh Token 已过期，请重新登录 |
| Refresh Token 已被使用 | 401 | LAK-01-008 | Refresh Token 已被使用（轮换检测），请重新登录 |

## 4. 对话接口详细设计

### 4.1 POST /api/v1/chat/message（核心接口）

#### 4.1.1 接口说明

发送对话消息，是系统的核心接口。**必需 JWT 认证**。支持两种响应模式：

- **普通模式**（`Accept: application/json`）：等待完整回答后返回 JSON
- **流式模式**（`Accept: text/event-stream`）：SSE 流式逐片返回回答

#### 4.1.2 Request Body

```json
{
  "sessionId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "message": "请问行政诉讼的时效是多久？"
}
```

#### 4.1.3 Request Schema

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|---------|------|
| sessionId | String | 否 | UUID 格式，若为空则创建新会话 | 会话 ID，为空则系统自动创建新会话 |
| message | String | 是 | 1-2000 字符，不能为空白 | 用户输入的消息文本 |

**字段校验规则**：

- `message` 不能为空字符串或纯空白，长度 1-2000 字符
- `sessionId` 若提供必须为合法 UUID 格式（`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）
- `message` 会经过敏感词过滤（双向校验：前缀树 + 大模型语义识别），命中后根据类型决定拦截或脱敏

#### 4.1.4 普通模式 Response（Accept: application/json）

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "answer": "根据《中华人民共和国行政诉讼法》第四十六条，公民、法人或者其他组织直接向人民法院提起诉讼的，应当自知道或者应当知道作出行政行为之日起六个月内提出。法律另有规定的除外。\n\n您是否想进一步了解：\n1. 行政诉讼的前置程序（行政复议）？\n2. 特殊类型的诉讼时效规定？\n3. 诉讼时效的中断和中止情形？",
    "sources": [
      {
        "docId": "DOC-2026-00123",
        "title": "中华人民共和国行政诉讼法（2026版）",
        "chunk": "第四十六条 公民、法人或者其他组织直接向人民法院提起诉讼的，应当自知道或者应当知道作出行政行为之日起六个月内提出。法律另有规定的除外。",
        "confidence": 0.95,
        "page": 12
      },
      {
        "docId": "DOC-2026-00124",
        "title": "XX市行政诉讼程序规定",
        "chunk": "第二条 本市各级人民法院受理行政诉讼案件，适用本规定。",
        "confidence": 0.87,
        "page": 3
      }
    ],
    "sessionId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "intentType": "POLICY_CONSULT",
    "confidence": 0.95,
    "tokens": 356,
    "ticketNo": null
  },
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### 4.1.5 普通模式 Response Schema

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| data.answer | String | 是 | AI 回答内容（Markdown 格式） |
| data.sources | Array | 否 | 溯源文档列表，为空表示未引用知识库 |
| data.sources[].docId | String | 是 | 文档唯一标识 |
| data.sources[].title | String | 是 | 文档标题 |
| data.sources[].chunk | String | 是 | 引用的原文片段 |
| data.sources[].confidence | Number | 是 | 该片段的检索置信度，范围 0.00-1.00 |
| data.sources[].page | Integer | 否 | 页码（若来源为文档） |
| data.sessionId | String | 是 | 会话 ID（新建或原有） |
| data.intentType | String | 是 | 意图类型枚举值 |
| data.confidence | Number | 是 | 意图识别置信度 0.00-1.00 |
| data.tokens | Integer | 否 | 本次回答消耗的 Token 数 |
| data.ticketNo | String | 否 | 工单编号（TICKET_SUBMIT 时返回） |
| data.extra | Object | 否 | 槽位状态（COLLECT_INFO 时必返回，见 §4.1.8） |

**extra 字段说明**（仅 COMPLAINT_SUGGEST / COLLECT_INFO 场景）：

| 字段 | 类型 | 说明 |
|------|------|------|
| extra.state | String | `COLLECT_INFO` / `TICKET_SUBMIT` / `CLOSED` |
| extra.slotStage | String | `started` / `fill` / `modify` / `interrupted` / `redirect` / `done` |
| extra.ticketNo | String | 工单编号（state=TICKET_SUBMIT 时） |
| extra.redirectToManual | Boolean | 是否引导用户转手工填写 |
| extra.prefilledSlots | Object | 已填槽位数据（redirectToManual=true 时携带） |
| extra.interruptReason | String | 中断原因（slotStage=interrupted 时） |

#### 4.1.8 COMPLAINT_SUGGEST 响应示例

**进入槽位采集**（state=COLLECT_INFO, slotStage=started）：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "sessionId": "...",
    "answer": "请问您要反馈的是哪类问题？\n1.治安投诉\n2.窗口服务投诉\n3.派出所/民警投诉\n4.其他",
    "intentType": "COMPLAINT_SUGGEST",
    "confidence": 1.0,
    "extra": { "state": "COLLECT_INFO", "slotStage": "started" }
  }
}
```

**工单创建成功**（state=TICKET_SUBMIT）：
```json
{
  "data": {
    "answer": "您的工单已创建，编号：TK-20260623-XXXXXXXX。我们会尽快处理，请留意反馈。",
    "intentType": "COMPLAINT_SUGGEST",
    "confidence": 1.0,
    "extra": { "ticketNo": "TK-20260623-XXXXXXXX", "state": "TICKET_SUBMIT" }
  }
}
```

**引导转手工填写**（slotStage=redirect）：
```json
{
  "data": {
    "answer": "对话已超过5轮，建议您转到页面继续填写。\n\n请在投诉建议页面继续填写，已填信息将自动带入。",
    "extra": { "state": "COLLECT_INFO", "slotStage": "redirect",
               "redirectToManual": true,
               "prefilledSlots": { "complaintType": "治安投诉", "contactName": "张三" } }
  }
}
```

**intentType 枚举值**：

| 枚举值 | 说明 | 路由目标 |
|--------|------|---------|
| `POLICY_CONSULT` | 政策咨询 | 子AgentA（政策咨询） |
| `PROCEDURE_GUIDE` | 办事指引 | 子AgentB（办事指引） |
| `COMPLAINT_SUGGEST` | 投诉建议 | 子AgentC（投诉建议） |
| `CHITCHAT` | 闲聊/无关 | 兜底处理 |
| `UNKNOWN` | 无法识别 | 兜底处理 |

#### 4.1.6 错误场景

| 触发条件 | HTTP 状态码 | errorCode | message |
|---------|------------|-----------|---------|
| 消息命中敏感词 | 403 | LAK-02-101 | 消息包含敏感内容，请调整后重试 |
| 请求频率过高（限流） | 429 | LAK-02-102 | 请求过于频繁，请稍后再试 |
| 大模型调用超时 | 504 | LAK-02-103 | 系统处理超时，请稍后重试 |
| 大模型不可用（降级） | 503 | LAK-99-501 | 当前服务不可用，请稍后重试 |
| 参数校验失败 | 400 | LAK-98-601 | 参数校验失败 |
| 会话不存在或已关闭 | 404 | LAK-02-104 | 会话不存在或已关闭 |
| 消息长度超限 | 400 | LAK-02-105 | 消息长度超过 2000 字符限制 |

#### 4.1.7 SSE 流式模式响应（Accept: text/event-stream）

SSE 流式模式（详见第 8 章 SSE 流式输出规范）用于逐片返回 AI 回答，提升用户感知的响应速度。

**时序说明**：

1. 前端发送 `POST /api/v1/chat/message` 请求，`Accept: text/event-stream`
2. 服务端立即返回 HTTP 200，`Content-Type: text/event-stream`
3. 服务端开始发送 SSE 事件流（事件类型见第 8 章）
4. 流结束后关闭连接

## 7. 错误码体系

### 7.1 错误码格式

所有错误码使用统一格式：`LAK-XX-XXX`（阿里巴巴Java开发手册推荐的 5 位数字错误码）

- **XX**：2 位数字模块编码
- **XXX**：3 位数字序号，模块内唯一

**模块编码对照**：

| 模块编码 | 模块名称 | 说明 |
|---------|---------|------|
| 01 | AUTH | 认证鉴权 |
| 02 | CHAT | 对话服务 |
| 03 | TICKET | 工单服务 |
| 98 | VALIDATION | 参数校验 |
| 99 | SYSTEM | 系统通用 |

### 7.2 错误码总表

#### 7.2.1 AUTH 模块（001-099）

| errorCode | HTTP 状态码 | message | 触发条件 |
|-----------|------------|---------|---------|
| LAK-01-001 | 401 | 未认证，请先登录 | 请求未携带 JWT 或 JWT 无效 |
| LAK-01-002 | 401 | 用户名或密码错误 | 登录时用户名不存在或密码不匹配 |
| LAK-01-003 | 403 | 账户已被禁用，请联系管理员 | 用户状态为 `DISABLED` 或 `LOCKED` |
| LAK-01-004 | 400 | 验证码错误 | 登录时验证码输入不正确 |
| LAK-01-005 | 400 | 验证码已过期，请重新获取 | 验证码超过 5 分钟有效期 |
| LAK-01-006 | 401 | 无效的 Refresh Token | Refresh Token 格式或签名无效 |
| LAK-01-007 | 401 | Refresh Token 已过期，请重新登录 | Refresh Token 超过 7 天有效期 |

| LAK-02-103 | 504 | 系统处理超时，请稍后重试 | 大模型调用超过超时阈值 |
| LAK-02-104 | 404 | 会话不存在或已关闭 | 请求的 sessionId 不存在或 status=CLOSED |
| LAK-02-105 | 400 | 消息长度超过 {limit} 字符限制 | 消息超过 2000 字符上限 |
| LAK-02-106 | 403 | 无权访问此会话 | 用户尝试访问/操作非自己创建的会话 |

#### 7.2.3 TICKET 模块（200-299）

| errorCode | HTTP 状态码 | message | 触发条件 |
|-----------|------------|---------|---------|
| LAK-03-201 | 409 | 该会话已创建过工单 | 同一 sessionId 重复创建工单 |
| LAK-03-202 | 500 | 工单提交失败，请稍后重试 | 工单适配器调用外部系统失败 |
| LAK-03-203 | 404 | 工单不存在 | 查询的 ticketNo 不存在 |
| LAK-03-204 | 403 | 无权访问此工单 | 用户尝试访问非自己创建的工单 |

#### 7.2.4 SYSTEM 模块（500-599）

| errorCode | HTTP 状态码 | message | 触发条件 |
|-----------|------------|---------|---------|
| LAK-99-500 | 500 | 系统内部错误 | 未捕获的服务器异常 |
| LAK-99-501 | 503 | 当前服务不可用，请稍后重试 | 大模型 API 调用失败/不可用（降级） |
| LAK-99-502 | 500 | 敏感词库加载失败，请检查配置 | 敏感词热加载时读取配置失败 |
| LAK-99-503 | 500 | 数据库操作异常 | Mybatis SQL 执行异常 |

#### 7.2.5 VALIDATION 模块（600-699）

| errorCode | HTTP 状态码 | message | 触发条件 |
|-----------|------------|---------|---------|
| LAK-98-601 | 400 | 参数校验失败 | 请求参数不满足校验规则（data 中返回具体字段） |
| LAK-98-602 | 400 | 请求体格式错误 | JSON 解析失败或 Content-Type 不匹配 |

### 7.3 错误码使用原则

1. **安全优先**：用户不存在和密码错误返回相同消息，避免枚举用户名
2. **最小信息**：生产环境错误消息不暴露内部实现细节（堆栈、SQL、文件路径等）
3. **分级处理**：
   - 客户端错误（4xx）：由前端展示给用户
   - 服务端错误（5xx）：由前端展示通用提示，同时后端记录完整异常信息到审计日志和监控系统

## 8. SSE 流式输出规范

### 8.1 概述

`POST /api/v1/chat/message` 接口在 `Accept: text/event-stream` 模式下使用 Server-Sent Events (SSE) 协议进行流式输出。客户端通过 `EventSource` 或 Fetch API 消费流。

### 8.2 连接建立

1. 客户端发送 POST 请求，包含 `Accept: text/event-stream` 头
2. 服务端校验参数和认证通过后返回 HTTP 200，设置以下响应头：

```
HTTP/1.1 200 OK
Content-Type: text/event-stream; charset=utf-8
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no
```

3. 服务端开始逐片发送 SSE 事件

### 8.3 事件类型定义

#### 8.3.1 event: message（文本片段）

```text
event: message
data: {"content":"根据《中华人民共和国行政诉讼法》第四十六条","tokenCount":15,"sessionId":"a1b2c3d4-e5f6-7890-abcd-ef1234567890","intentType":"POLICY_CONSULT","index":0}
```

**data JSON Schema**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| content | String | 是 | AI 回答的文本片段 |
| tokenCount | Integer | 是 | 该片段的 Token 数 |
| sessionId | String | 是 | 会话 ID |
| intentType | String | 否 | 意图类型（仅在首片消息中返回） |
| index | Integer | 是 | 片段序号，从 0 开始递增 |

#### 8.3.2 event: done（流结束）

```text
event: done
data: {"sessionId":"a1b2c3d4-e5f6-7890-abcd-ef1234567890","sources":[{"docId":"DOC-2026-00123","title":"中华人民共和国行政诉讼法（2026版）","chunk":"第四十六条...","confidence":0.95,"page":12}],"intentType":"POLICY_CONSULT","confidence":0.95,"tokens":356,"ticketNo":null}
```

**data JSON Schema**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| sessionId | String | 是 | 会话 ID |
| sources | Array | 否 | 溯源文档列表 |
| intentType | String | 是 | 意图类型 |
| confidence | Number | 是 | 意图识别置信度 |
| tokens | Integer | 否 | 总 Token 消耗数 |
| ticketNo | String | 否 | 工单编号（TICKET_SUBMIT 时） |
| state | String | 否 | 槽位状态：`COLLECT_INFO` / `TICKET_SUBMIT` / `CLOSED` |
| slotStage | String | 否 | 当前槽位阶段：`started` / `fill` / `modify` / `interrupted` / `redirect` / `done` |
| redirectToManual | Boolean | 否 | 是否引导转手工填写 |
| prefilledSlots | Object | 否 | 已填槽位键值对（redirectToManual 时携带） |
| extra | Object | 否 | 槽位状态完整对象（建议优先使用顶层字段，extra 为兼容保留） |

**COMPLAINT_SUGGEST done 事件示例**：
```text
event: done
data: {"sessionId":"...","intentType":"COMPLAINT_SUGGEST","state":"COLLECT_INFO","slotStage":"fill","ticketNo":null}
```

#### 8.3.3 event: error（流中错误）

```text
event: error
data: {"errorCode":"LAK-02-101","message":"消息包含敏感内容，请调整后重试","sessionId":"a1b2c3d4-e5f6-7890-abcd-ef1234567890"}
```

**data JSON Schema**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| errorCode | String | 是 | 错误码 |
| message | String | 是 | 错误消息 |
| sessionId | String | 是 | 会话 ID |

### 8.4 完整 SSE 流示例

```text
event: message
data: {"content":"根据《中华人民共和国行政诉讼法》第四十六条","tokenCount":15,"sessionId":"a1b2c3d4-...","intentType":"POLICY_CONSULT","index":0}

event: message
data: {"content":"，公民、法人或者其他组织直接向人民法院提起诉讼的","tokenCount":18,"sessionId":"a1b2c3d4-...","index":1}

event: message
data: {"content":"，应当自知道或者应当知道作出行政行为之日起六个月内提出。","tokenCount":16,"sessionId":"a1b2c3d4-...","index":2}

event: done
data: {"sessionId":"a1b2c3d4-...","sources":[{"docId":"DOC-2026-00123","title":"中华人民共和国行政诉讼法（2026版）","chunk":"第四十六条...","confidence":0.95,"page":12}],"intentType":"POLICY_CONSULT","confidence":0.95,"tokens":356,"ticketNo":null}
```

### 8.5 连接超时与重连策略

| 策略项 | 配置值 | 说明 |
|--------|--------|------|
| 服务端空闲超时 | 60 秒 | 超过 60 秒未产生新片段则主动断开连接 |
| 客户端重连间隔 | 3 秒 | 客户端断连后等待 3 秒重连 |
| 最大重连次数 | 3 次 | 超过后提示用户刷新重试 |
| 重连标识 | `Last-Event-Id` | 断连重连时携带上次收到的最后一个 `index` 值 |

### 8.6 客户端消费示例（JavaScript）

```javascript
// 使用 Fetch API 消费 SSE 流（推荐方式，支持 POST）
async function sendMessage(sessionId, message, token) {
  const response = await fetch('/api/v1/chat/message', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Accept': 'text/event-stream',
      'Authorization': `Bearer ${token}`
    },
    body: JSON.stringify({ sessionId, message })
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop() || '';

    let eventType = '';
    for (const line of lines) {
      if (line.startsWith('event: ')) {
        eventType = line.slice(7);
      } else if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        switch (eventType) {
          case 'message':
            // 追加文本片段到显示区域
            appendAnswer(data.content);
            break;
          case 'done':
            // 流结束，显示溯源信息
            showSources(data.sources);
            break;
          case 'error':
            // 处理流中错误
            showError(data.message);
            break;
        }
      }
    }
  }
}
```

> **兼容性说明**：标准的 `EventSource` API 仅支持 GET 请求，而本接口为 POST 请求。因此推荐使用 `Fetch API` + `ReadableStream` 方式消费 SSE 流。如需兼容 `EventSource`，服务端需要额外提供一个 GET 端点轮询队列结果（P1 优化项）。
