## 2. Filter Chain 拦截链路

### 2.1 执行顺序与职责

| Order | 组件 | 类型 | 职责 |
|-------|------|------|------|
| 1 | `SensitiveWordPreCheckFilter` | `Filter` | 用户输入敏感词前置拦截 |
| 2 | `TraceIdFilter` | `Filter` | 生成/提取 TraceId 注入 MDC |
| 3 | `AuditLogInterceptor` | `HandlerInterceptor` | 请求体捕获 + 入站审计 |
| 4 | `AuthFilter` | `Filter` | JWT Token 校验 + 用户身份注入 |
| 5 | `RateLimitFilter` | `Filter` | 基于用户/IP 的请求频率限制 |
| 6 | Spring `RequestMapping` | 框架路由 | Controller 方法分发 |

### 2.2 设计决策：Filter vs Interceptor

| 组件 | 实现方式 | 理由 |
|------|---------|------|
| 敏感词前置 | `Filter` | 最早执行，在请求体被业务代码消费前完成校验 |
| TraceId | `Filter` | 最先注入 MDC，确保所有下游日志带 TraceId |
| 审计入站 | `HandlerInterceptor` | 可访问 Spring Bean (AuditLogMapper)，且 preHandle → afterCompletion 可计算耗时 |
| JWT认证 | `Filter` | 在 Interceptor 之前执行，保证审计拦截器的 afterCompletion 能拿到 userId |
| 限流 | `Filter` | 在 Interceptor 之前执行，被限流的请求不写入审计日志 |

### 2.3 白名单端点（免认证）

以下端点不经过 `AuthFilter`，但经过其他所有 Filter：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/v1/health` | 健康检查 |
| `POST` | `/api/v1/auth/login` | 用户登录 |
| `GET` | `/api/v1/auth/captcha` | 获取验证码 |
| `POST` | `/api/v1/auth/refresh` | 刷新 Token |

### 2.4 审计拦截器跳过路径

以下路径不写入审计日志（避免噪声）：

```
/health, /actuator, /api/v1/health, /error
```

## 3. 认证机制

### 3.1 JWT Token 体系

| 参数 | 值 |
|------|-----|
| 签名算法 | HMAC-SHA256 (JJWT 库) |
| Access Token 有效期 | 7200s (2小时) |
| Refresh Token 有效期 | 604800s (7天) |
| 签发者 (iss) | `lak-ai-platform` |

**Access Token Payload**：
```json
{
  "iss": "lak-ai-platform",
  "sub": "admin",
  "userId": 1,
  "roles": ["ADMIN"],
  "iat": 1687345200,
  "exp": 1687352400
}
```

**Refresh Token Payload**：
```json
{
  "iss": "lak-ai-platform",
  "sub": "admin",
  "type": "refresh",
  "userId": 1,
  "iat": 1687345200,
  "exp": 1687950000
}
```

### 3.2 AuthFilter 处理流程

```
1. 检查请求路径是否在白名单 → 是则放行
2. 提取 Authorization: Bearer <token>
3. 解析 JWT → 验证签名 + 有效期
4. 成功 → 注入 request attributes:
   - userId (Long)
   - username (String)
   - roles (List<String>)
5. 失败 → 响应 JSON:
   - Token缺失: {"code": "LAK-01-001", "message": "未认证"}       (401)
   - Token过期: {"code": "LAK-01-006", "message": "Token已过期"}  (401)
```

### 3.3 Token 防重放

Refresh Token 采用**滚动轮换 + 旧Token撤销**策略：

```
1. 用户调用 /auth/refresh 提交 refreshToken
2. 服务端解析验证通过后:
   a. 计算旧 refreshToken 的 SHA-256 哈希
   b. 写入 Redis: token:revoked:<SHA-256> = "1", TTL = 剩余有效期
   c. 签发新的 accessToken + refreshToken 对
3. 下次请求若仍使用旧 refreshToken → Redis 命中 → 拒绝
```

### 3.4 登录流程

```
POST /api/v1/auth/login {username, password, captchaKey, captchaCode}
  │
  ├─ 1. 验证码校验 → CaptchaService.validate(captchaKey, captchaCode)
  │    失败 → LAK-01-004（验证码错误）/ LAK-01-005（验证码过期）
  │
  ├─ 2. 用户查询 → SELECT * FROM sys_user WHERE username = ?
  │    不存在 → LAK-01-002（用户名或密码错误）— 不区分用户不存在和密码错误
  │
  ├─ 3. 密码验证 → BCrypt.matches(rawPassword, passwordHash)
  │    失败 → LAK-01-002
  │
  ├─ 4. 状态检查 → status == ACTIVE ?
  │    DISABLED → LAK-01-003（账户已禁用）
  │    LOCKED   → LAK-01-003（账户已锁定）
  │
  ├─ 5. 查询角色 → SELECT role_code FROM sys_role WHERE id = role_id
  │
  └─ 6. 签发 Token → 返回 {accessToken, refreshToken, expiresIn, userInfo}
```

> **安全设计**：步骤 2+3 返回相同的错误码和错误消息（"用户名或密码错误"），防止用户枚举攻击。

### 3.5 验证码服务

| 参数 | 值 |
|------|-----|
| 生成算法 | `SecureRandom` 4位数字 |
| 存储 Key | `captcha:<16-char-uuid>` |
| TTL | 300s (5分钟) |
| 校验方式 | `getAndDelete()`（一次性，防止重放） |
| 大小写 | 不敏感 |

**当前实现**：开发环境返回明文验证码文本。**生产环境必须替换**为 captchaKey + base64 PNG 图片，前端展示图片而非明文。

### 3.6 密码安全

| 项目 | 说明 |
|------|------|
| 哈希算法 | BCrypt (Spring Security `PasswordEncoder`) |
| 加盐方式 | 算法内置自动加盐 |
| 禁止算法 | MD5 / SHA1 / SHA256 裸哈希 |

## 4. 授权机制

### 4.1 RBAC 模型

系统采用三层权限模型：

```
sys_user (用户)
  │ N:1
  ▼
sys_role (角色)
  │ N:M (via sys_role_permission)
  ▼
sys_permission (权限 — 预留扩展)
  │
  └── 当前阶段：权限主要通过 Menu 体系控制
```

**预置角色**：

| 角色 | role_code | 权限范围 |
|------|-----------|---------|
| 管理员 | `ADMIN` | 全部菜单 + 全部接口 |
| 操作员 | `OPERATOR` | 知识库管理 + 工单处理 |
| 普通用户 | `USER` | 对话 + 工单 |

### 4.2 Menu-Based 权限控制

由于前端通过菜单渲染控制页面可见性，后端通过 `MenuPermissionChecker` 做接口级权限校验：

```java
// 典型用法
@GetMapping("/admin/audit-logs")
public ApiResponse<...> list(..., HttpServletRequest request) {
    menuPermissionChecker.requireMenu(request, "audit");  // 检查用户是否有 audit 菜单权限
    // ...业务逻辑
}
```

**检查逻辑**：

```
1. 从 request attribute 获取 roles
2. 若 roles 包含 "ADMIN" → 直接通过
3. 否则:
   a. SELECT id FROM sys_menu WHERE menu_code = ?
   b. SELECT 1 FROM sys_role_menu WHERE role_id = ? AND menu_id = ?
   c. 存在记录 → 通过 / 不存在 → throw AuthException("权限不足")
```

### 4.3 菜单-权限映射

当前预置菜单：

| menu_code | 菜单名称 | 父菜单 | 对应 Controller |
|-----------|---------|--------|----------------|
| `chat` | 智能问答 | — | ChatController |
| `ticket` | 工单管理 | — | TicketController |
| `admin` | 系统管理 | — | AdminView |
| `knowledge` | 知识库管理 | `admin` | KnowledgeController |
| `sensitive` | 敏感词管理 | `admin` | AdminController |
| `role` | 角色管理 | `admin` | RoleManageController |
| `audit` | 操作审计 | `admin` | AuditLogController |

## 5. 敏感词双向校验

### 5.1 拦截点布局

```
用户提问 ──▶ [前置拦截] SensitiveWordPreCheckFilter
                │  命中 → 403 + "消息包含敏感内容，请调整后重试"
                │  （不泄露具体命中了哪个词）
                ▼
           业务处理 (Agent + RAG + LLM)
                │
                ▼
           [后置拦截] ComplianceValidator
                │  命中 → 替换为"系统繁忙，请稍后重试"
                │       记录告警日志
                ▼
           返回用户
```

### 5.2 敏感词库

| 属性 | 说明 |
|------|------|
| 存储位置 | `classpath:config/sensitive-words.txt` |
| 格式 | 每行一个词，`#` 开头为注释 |
| 加载时机 | 应用启动时 + `POST /admin/sensitive-words/reload` 热加载 |
| 线程安全 | `volatile` 引用替换，`ConcurrentHashMap` 存储 |
| 匹配算法 | `String.contains(word)` — 子串匹配 |

### 5.3 敏感词前置拦截实现

```java
// SensitiveWordPreCheckFilter — 仅拦截 POST /api/v1/chat/message
public void doFilter(request, response, chain) {
    // 1. 使用 CachedBodyHttpServletRequestWrapper 缓存请求体
    // 2. 读取请求体提取消息文本
    // 3. containsSensitiveWord(message)
    //    → 遍历 ConcurrentHashMap，text.contains(word)
    // 4. 命中 → return 403 JSON error (不提示具体词汇)
    // 5. 未命中 → chain.doFilter(wrappedRequest, response)
}
```

### 5.4 敏感词后置校验实现

```java
// ComplianceValidator — AI 答复生成后执行
public boolean validate(String answer, List<RagFragment> fragments, boolean isFallback) {
    if (answer == null || answer.isBlank()) return false;
    if (answer.length() < 5) return false;
    if (sensitiveWordFilter.containsSensitiveWord(answer)) return false;
    return true;
}
```

### 5.5 热加载

```
POST /api/v1/admin/sensitive-words/reload  (需 JWT + ADMIN 角色)
  → SensitiveWordPreCheckFilter.reload()
    → 从 classpath:config/sensitive-words.txt 重新读取
    → 构建新的 ConcurrentHashMap
    → volatile 替换引用（无锁切换）
    → 返回 {"count": N}
```

### 5.6 安全设计要点

| 原则 | 实现 |
|------|------|
| **不泄露命中词** | 错误响应为通用消息，不包含具体敏感词 |
| **双向校验** | 用户输入 + AI 答复均校验，防止两端漏过 |
| **热加载无中断** | volatile 引用替换，加载过程中继续使用旧词库 |
| **违规必记录** | 含敏感词的答复在日志中以 WARN 级别记录（含 sessionId），但不返回给用户 |

## 6. 审计日志体系

### 6.1 三层采集机制

```
┌─────────────────────────────────────────────────┐
│ Layer 1: AuditLogInterceptor (HandlerInterceptor)│
│ - 覆盖: 所有 /api/** 请求（除白名单）             │
│ - 采集: traceId, userId, requestUri, latencyMs   │
│ - 时机: preHandle(暂存) → afterCompletion(写入)  │
├─────────────────────────────────────────────────┤
│ Layer 2: AuditLogAspect (AOP @Around)            │
│ - 覆盖: @AuditLog 注解的方法                      │
│ - 采集: traceId, status, latencyMs               │
│ - 目标: ChatController.sendMessage()             │
│         TicketController.createTicket()           │
├─────────────────────────────────────────────────┤
│ Layer 3: 业务代码手动注入 (未来扩展)               │
│ - 采集: requestBody, responseBody, modelParams   │
│         modelResponse, retrievalFragments        │
│ - 通过 Mybatis-Plus AuditLogMapper.insert()      │
└─────────────────────────────────────────────────┘
```

### 6.2 审计日志表

| 属性 | 说明 |
|------|------|
| 表名 | `audit_log_yyyyMM`（如 `audit_log_202606`） |
| 分表策略 | 按月分表，Mybatis-Plus 动态表名拦截器自动路由 |
| 权限 | 仅 INSERT + SELECT，禁止 UPDATE / DELETE |
| 留存 | 6 个月在线 + 归档文件长期保留 |
| 存储引擎 | InnoDB |

### 6.3 审计字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `trace_id` | VARCHAR(64) | 全链路追踪 ID |
| `session_id` | VARCHAR(64) | 关联会话 ID |
| `user_id` | BIGINT | 操作用户 ID |
| `request_body` | MEDIUMTEXT | 请求体 / 用户输入 |
| `response_body` | MEDIUMTEXT | 响应体 / AI 答复 |
| `intent_type` | VARCHAR(32) | 意图类型 |
| `confidence` | DECIMAL(3,2) | 置信度 |
| `model_params` | JSON | 大模型调用参数 |
| `model_response` | MEDIUMTEXT | 大模型原始返回 |
| `retrieval_fragments` | JSON | RAG 检索片段 |
| `latency_ms` | INT | 全链路耗时 |
| `status` | VARCHAR(16) | SUCCESS / FAIL / FALLBACK |
| `error_message` | TEXT | 异常信息 |
| `create_time` | DATETIME | 记录时间 |

### 6.4 审计日志防篡改

| 措施 | 实现 |
|------|------|
| **禁止修改** | 数据库用户仅持有 `SELECT, INSERT` 权限于 `audit_log_*` 表 |
| **禁止删除** | 同上，无 `DELETE` 权限 |
| **分表隔离** | 按月分表，即使单表被破坏，其他月份数据完整 |
| **TraceId 关联** | 同一次请求的所有日志（框架日志 + 业务日志 + 审计日志）通过 TraceId 关联 |

### 6.5 审计查询安全

| 措施 | 实现 |
|------|------|
| **月份参数校验** | `Pattern.compile("^\\d{6}$")` 正则验证，防 SQL 注入 |
| **权限控制** | `MenuPermissionChecker.requireMenu(request, "audit")` 前置校验 |
| **不存在表格容错** | 查询不存在的分表时捕获 `DataAccessException`，返回空结果而非 500 错误 |
| **ThreadLocal 清理** | `MybatisPlusConfig.setAuditMonth()` 在 finally 中 `clearAuditMonth()`，防止线程池泄露 |

## 7. 限流保护

### 7.1 滑动窗口算法

基于 Redis Sorted Set + Lua 脚本实现原子操作：

```lua
-- rate_limit.lua
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)  -- 清理过期记录
local count = redis.call('ZCARD', key)                        -- 计算当前窗口请求数
if count >= limit then
    return 0  -- 拒绝
end
redis.call('ZADD', key, now, now .. '-' .. math.random())    -- 添加新记录
redis.call('EXPIRE', key, window)                             -- 刷新 TTL
return 1  -- 通过
```

### 7.2 限流参数

| 参数 | 默认值 | 配置键 |
|------|--------|--------|
| 每分钟最大请求数 | 100 (dev) / 30 (prod) | `lak.rate-limit.max-requests-per-minute` |
| 滑动窗口大小 | 60s | `lak.rate-limit.window-seconds` |
| Key 策略（已认证） | `rate_limit:{userId}:{apiPath}` | — |
| Key 策略（未认证） | `rate_limit:{ip}:{apiPath}` | — |
| Redis DB | db1 | — |
| 超限 HTTP 状态码 | 429 Too Many Requests | — |

### 7.3 限流异常

```json
{
  "code": "LAK-02-102",
  "message": "请求过于频繁，请稍后再试",
  "traceId": "..."
}
```

`RateLimitException` 由 `GlobalExceptionHandler` 统一映射为 HTTP 429。

## 9. 通信加密

### 9.1 加密端点

| 通信链路 | 协议 | 说明 |
|---------|------|------|
| 客户端 → Nginx | **HTTPS (TLS 1.2+)** | 生产环境 Nginx 终结 TLS |
| Nginx → lak-app | HTTP (内网) | 内网通信可选 HTTPS |
| lak-app → MySQL | TCP (内网) | 等保三级需启用 TLS |
| lak-app → Redis | TCP + AUTH | Redis 密码认证 |
| lak-app → Qdrant | gRPC/HTTP (内网) | 可选 API Key 认证 |
| lak-app → 百炼 API | **HTTPS** | 公网通信，百炼 API 默认 HTTPS |

## 10. 熔断与韧性

### 10.1 Resilience4j 配置

```yaml
resilience4j:
  circuitbreaker:
    instances:
      model-call:
        sliding-window-type: TIME_BASED
        sliding-window-size: 30          # 30秒窗口
        failure-rate-threshold: 50       # 50% 失败率触发熔断
        permitted-number-of-calls-in-half-open-state: 3
        wait-duration-in-open-state: 30s # 熔断后 30s 尝试半开
  retry:
    instances:
      model-call:
        max-attempts: 3                  # 最大 3 次重试
        wait-duration: 1s
        exponential-backoff-multiplier: 2  # 1s → 2s → 4s
        retry-exceptions:
          - java.net.SocketTimeoutException
          - java.io.IOException
```

### 10.2 降级策略

| 场景 | 降级行为 |
|------|---------|
| LLM 调用超时/失败 (Retry 3 次后) | CircuitBreaker 打开 → 返回"系统繁忙"兜底答复 |
| Qdrant 检索异常 | `HybridRetriever` 返回空列表 → Agent 以空上下文生成（合规校验会兜底） |
| Redis 不可用 | 会话/限流功能降级；AuthFilter 继续工作（JWT 自包含） |
| MySQL 不可用 | 全局 500 错误；审计日志丢失风险（需监控告警） |

## 11. 全局异常处理

### 11.1 异常映射

| 异常类 | HTTP 状态码 | 错误码 | 说明 |
|--------|-----------|--------|------|
| `AuthException` | 401 / 403 / 400 | LAK-01-xxx | 认证相关（根据具体错误映射） |
| `SensitiveWordException` | 403 | — | 敏感词拦截（不暴露详情） |
| `RateLimitException` | 429 | LAK-02-102 | 请求频率过高 |
| `ChatException` | 400 / 503 | LAK-02-xxx | 对话异常 |
| `TicketException` | 500 | LAK-03-xxx | 工单异常 |
| `ModelException` | 503 | LAK-99-501 | 大模型不可用 |
| `KnowledgeException` | 400/404/500 | — | 知识库异常 |
| `MethodArgumentNotValidException` | 400 | LAK-98-601 | 参数校验失败 |
| 未捕获 `Exception` | 500 | LAK-99-502 | 系统内部错误（兜底） |

### 11.2 统一响应格式

```json

  "code": 200,
  "message": "success",
  "data": { ... },
```

## 12. 输入安全

### 12.1 请求体缓存

`CachedBodyHttpServletRequestWrapper` — 缓冲整个 Input Stream，允许多个 Filter 重复读取请求体而无需 Controller 重复解析：

- `SensitiveWordPreCheckFilter` 读取检查敏感词
- `AuditLogInterceptor` 读取写入审计日志
- Controller 读取进行业务处理

### 12.2 消息长度限制

| 参数 | 值 |
|------|-----|
| 单条消息最大 | 2000 字符 (`lak.message.max-length`) |
| 文件上传最大 | 20MB (`spring.servlet.multipart.max-file-size`) |

### 12.3 路径穿越防护

`LocalFileStorageService` 所有文件操作前执行：

```java
// 归一化路径 + 验证最终路径在 rootDir 范围内
Path resolved = rootDir.resolve(relativePath).normalize();
if (!resolved.startsWith(rootDir)) {
    throw new SecurityException("路径穿越攻击检测");
}
```

|------|---------|

| 生产 | 通过 Nginx 同源代理，不启用 CORS |
